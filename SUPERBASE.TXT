3. AJUSTAR A TABELA auth.users COM profiles
O frontend novo busca nome em:
- public.profiles

Voce precisa garantir que cada usuario autenticado tenha um registro em:
- public.profiles

Campos importantes:
- id = mesmo UUID do auth.users.id
- full_name
- email
- phone opcional

Pode fazer manualmente em:
Table Editor
profiles

Ou via SQL, exemplo:
insert into public.profiles (id, full_name, email)
values ('UUID_DO_USER', 'Pedro', 'email@dominio.com');

4. VINCULAR O USUARIO A UMA UNIDADE E ROLE
Tabela:
- public.user_unit_roles

Sem isso:
- o login entra
- mas o usuario pode aparecer como "Sem perfil"
- e as areas protegidas podem negar acesso

Voce precisa inserir pelo menos 1 linha por usuario:
- user_id = UUID do usuario do auth
- unit_id = UUID da unidade
- role = admin ou attendant ou cashier

Exemplo SQL:
insert into public.user_unit_roles (user_id, unit_id, role)
values ('UUID_DO_USER', 'UUID_DA_UNIDADE', 'admin');

Observacao:
- hoje nao existe tabela units formal na migration
- entao o unit_id precisa ser um UUID consistente usado em todas as tabelas

5. GARANTIR unit_id NOS DADOS OPERACIONAIS
As policies RLS dependem de:
- auth.uid()
- user_unit_roles.unit_id
- unit_id nas tabelas operacionais

Entao, ao cadastrar dados manualmente no banco, inclua sempre o mesmo unit_id em:
- clients
- client_vehicles
- parking_spots
- vehicle_stays
- transactions

Se o unit_id nao bater com o do usuario logado:
- os dados nao aparecem
- vai parecer erro de permissao ou lista vazia
